# Security

→ See [RELIABILITY.md](RELIABILITY.md) for failure modes.

---

## Threat model

Knowledge projects handle potentially sensitive source material: internal
documents, proprietary research, personal data. The pipeline must not
silently exfiltrate or expose that material.

**In scope**:
- Source files stored in `sources/`
- Extracted content in `extractions/`
- Glossary terms and definitions
- Query log (which may contain sensitive questions)

**Out of scope**:
- Network-level security (the pipeline is local by default)
- Authentication and multi-user access control (v1 is single-user, local)

---

## Data handling rules

### Rule 1 — Sources never leave the local project directory

The pipeline does not upload source files to external services.
LLM API calls send only extracted text chunks, not raw files.
Chunks are sized to minimize what is sent in any single call.

### Rule 2 — API keys are never written to project files

`.knowledge-project`, `.meta.json`, and all generated files must not
contain API keys, tokens, or credentials. These are read from environment
variables only.

### Rule 3 — `kb/questions/` may contain sensitive intent

Question files reveal what the user is investigating. For most internal
knowledge projects this is fine to version — the compound knowledge value
depends on keeping history. However, for sensitive projects (competitive
intelligence, legal matters) the questions directory should be excluded from
public repositories.

`init` does not exclude `kb/questions/` by default. Use `init --private-queries`
to add it to `.gitignore`, or add it manually after init.

### Rule 4 — Generated files are not authoritative for compliance

Extracted summaries and glossary definitions are LLM-generated and may
contain errors. They must not be used for compliance, legal, or medical
decisions without human review.

---

## Sensitive source handling

When a source is flagged as sensitive (via `ingestion add --sensitive`):

- Its `source-id` directory is added to `.gitignore` automatically.
- The extraction JSON is also excluded from git.
- The source is still processed normally; the flag only affects version control.
